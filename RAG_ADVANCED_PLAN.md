# RAG 进阶与代理化路线图

本路线图用于承接已完成的基础 RAG 优化工作，目标是将当前系统从“可用的检索增强问答”推进到“可演进、可观测、可扩展的 Agentic RAG 平台”。

已完成的基础优化见 [RAG_OPTIMIZATION_PLAN(completed).md](RAG_OPTIMIZATION_PLAN(completed).md)。本文聚焦后续能力建设，不重复记录已完成项。

## 1. 总体目标

### 1.1 业务目标

- 提高复杂文档场景下的解析质量，减少“抽取错位、结构丢失、表格失真”等问题。
- 提高复杂问题场景下的召回率、相关性和答案忠实度。
- 让系统具备从“单轮检索问答”向“可规划、可回退、可调用工具”的 Agentic 工作流演进的基础。
- 为后续评测、灰度、反馈闭环和持续优化建立稳定的数据面与控制面。

### 1.2 技术目标

- 文档输入从 `File -> Plain Text -> Chunks` 升级为 `File -> Structured Segments -> Chunks -> Retrieval Units`。
- 检索从“单策略混合检索”升级为“多查询、多粒度、可重排、可解释检索”。
- 生成从“单次回答”升级为“带引用、可验证、可追踪回答”。
- 观测从“基础 trace”升级为“可评测、可对比、可回归”的实验体系。

### 1.3 北极星指标

建议统一使用以下指标作为阶段验收口径：

| 类别 | 指标 | 目标方向 |
|---|---|---|
| 解析质量 | 表格还原准确率、标题层级准确率、阅读顺序准确率 | 持续提升 |
| 检索质量 | Recall@K、MRR、citationHitRate、passRate | 持续提升 |
| 生成质量 | Faithfulness、Answer Relevance、人工反馈正向率 | 持续提升 |
| 性能 | 单文档解析耗时、单次问答延迟、重排耗时占比 | 可控下降 |
| 成本 | 单文档 embedding 成本、单 query 总成本 | 可控下降 |

## 2. 当前基线

当前系统已经具备以下基础能力：

- 混合检索与 RRF 融合
- Query Rewrite / Multi-Query 基础能力
- 可切换 Reranker 端口与适配器骨架
- 按知识库覆盖检索阈值、`vectorTopK`、`keywordTopK`
- PromptTemplate 与基础 Trace / Cost 统计
- 异步文档处理、分阶段耗时审计、增量 embedding 复用基础

因此，后续工作重点不再是“把基础链路做通”，而是围绕“结构化输入、高质量检索、Agentic 流程、闭环评测”继续向前推进。

## 3. 分阶段路线图

## 第一阶段：文档智能解析与语义闭环

### 3.1 目标

建立结构化文档理解能力，让解析结果不再只是纯文本，而是具备版面、层级、表格、图像、语义片段等信息，为后续分块、检索和溯源提供高质量输入。

### 3.2 里程碑

#### 3.2.1 布局感知解析

- 引入 DeepDOC 风格的视觉解析流水线。
- 产出页面级 `LayoutBlock`、`LogicalSegment`、`ReadingOrder`。
- 支持正文、标题、表格、图片、页眉页脚、脚注等块类型识别。
- 详细设计见 [docs/DEEPDOC_DESIGN.md](docs/DEEPDOC_DESIGN.md)。

#### 3.2.2 多场景专业解析器

建立 `ParserKey(extension, strategy)` 分派体系，支持以下策略：

| 策略 | 适用场景 | 关键能力 |
|---|---|---|
| `naive` | 默认兜底 | 基础提取与简单分块 |
| `paper` | 论文、技术报告 | 双栏检测、摘要识别、章节层级 |
| `manual` | 手册、规范文档 | 目录树、书签、大纲边界 |
| `laws` | 法律法规 | 编章节目条树状结构 |
| `qa` | FAQ、帮助中心 | 问答配对、条目对齐 |
| `table` | 报表、台账 | 表格结构化、列语义推断 |
| `resume` | 简历 | 字段抽取、时间线重组 |
| `email` | 邮件 | multipart、附件递归 |

#### 3.2.3 语义分块与层级合并

- 从“按字符/段落切块”升级为“结构感知 + Token 感知 + 场景感知”分块。
- 为每个 chunk 补充 `headingPath`、`segmentType`、`keywords`、`docType` 等元数据。
- 支持“父块存储 + 子块检索”的 small-to-big 基础结构。

#### 3.2.4 增量更新与生命周期管理

- 文档重处理时仅重建变化 chunk。
- 索引和元数据建立版本号与时间戳。
- 支持知识库重建、文档覆盖、软删除清理、陈旧 chunk 回收。

### 3.3 验收标准

- 至少支持 `pdf + paper/manual/table` 三类场景的结构化解析落地。
- 文档入库结果包含标题路径、块类型和来源定位信息。
- 对同一文档重复处理时，可复用未变化内容的 embedding。
- 前端或调试接口能够展示结构化解析结果，而非仅展示纯文本。

### 3.4 风险与依赖

- OCR、Layout、Table 模型体积大，对部署环境要求更高。
- Java 侧视觉流水线需要在性能与依赖体积之间做平衡。
- 复杂 PDF 的标注样本不足时，策略规则需要先行兜底。

## 第二阶段：高性能检索

### 4.1 目标

把检索层从“能搜到”提升到“搜得准、搜得全、搜得稳、可解释”。

### 4.2 里程碑

#### 4.2.1 混合检索深化

- 完善 `pgvector + PostgreSQL FTS/BM25` 的双路检索参数体系。
- 对知识库维度开放阈值、topK、重排开关、缓存等配置。
- 支持 A/B 比较不同融合与重排策略。

#### 4.2.2 父子文档检索

- 子 chunk 负责召回，父 chunk 或段落簇负责生成上下文。
- 解决命中内容太碎、上下文断裂的问题。
- 支持“子块命中，自动拼接前后相邻块”与“向上回溯父块”的两种模式。

#### 4.2.3 动态重排

- 根据 query 类型、初始召回质量、历史 trace 统计动态调整候选数。
- 将 `CrossEncoderRerankerAdapter` 的输入控制在稳定成本范围内。
- 支持“快速词法重排 -> 精排 cross-encoder”两级模式。

### 4.3 验收标准

- 在同一评测集上，召回率与引用命中率相较基础版本有稳定提升。
- 对长文档复杂问答，答案引用片段的完整性明显改善。
- 检索 trace 中可清晰看到 query、召回、融合、重排、截断各阶段数据。

## 第三阶段：Agentic 工作流

### 5.1 目标

让系统从“单步检索问答”升级为“会规划、会修正、会用工具”的智能流程。

### 5.2 里程碑

#### 5.2.1 查询变换

- Multi-Query：从不同角度扩展问题，覆盖更多召回维度。
- Query Decomposition：将复杂问题拆成子问题后分别检索。
- HyDE：生成假设答案后再做检索。
- Step-Back：先抽象问题，再联合原始问题检索。

#### 5.2.2 Self-RAG

- 增加检索证据质量评估步骤。
- 当证据不足时触发二次检索、回退查询或限制回答。
- 对“高置信回答”和“证据不足拒答”建立显式判定逻辑。

#### 5.2.3 Tool Use

- Agent 根据问题类型决定走知识库、外部搜索、计算工具或混合路径。
- 为每个工具调用写入 trace 节点，支持完整回放。
- 约束工具使用成本和权限，避免无限扩张为通用代理。

### 5.3 验收标准

- 对复合问题能拆解出可解释的中间检索步骤。
- 对知识库中不存在的内容能优先拒答或回退，而不是直接幻觉生成。
- 工具调用路径在日志和前端调试面板中可观察。

## 第四阶段：工业级 UX 与闭环评测

### 6.1 目标

把技术能力转化为用户可感知的交互体验，并建立稳定的质量回归体系。

### 6.2 里程碑

#### 6.2.1 流式渲染与深度溯源

- 后端提供 SSE/流式事件输出。
- 前端按 token 或段落增量渲染回答。
- 引用支持回跳到具体文档、页码、块坐标或高亮区域。

#### 6.2.2 自动化评测

- 集成 RAGas 或等价自建指标。
- 建立标准评测集、回归基线、版本对比报告。
- 对 rerank、rewrite、chunking、parser 等关键策略支持独立评测。

#### 6.2.3 反馈闭环

- 接入用户点赞/点踩、补充反馈、纠错标记。
- 低分样本自动沉淀到评测集候选池。
- 为后续 prompt、retrieval、parser 优化提供真实世界样本。

### 6.3 验收标准

- 用户可以看到回答生成过程和引用来源。
- 每次策略变更后都有回归指标，不再依赖主观判断上线。
- 线上低分问题可沉淀为可复现样本。

## 7. 建议实施顺序

建议按下面顺序推进，避免“上层能力建设快于底层数据质量”：

1. 完成结构化解析最小闭环：`pdf/manual/paper` 三类场景 + headingPath + table 结构化。
2. 完成 small-to-big 检索与动态重排，让长文档问答先变稳。
3. 补齐 Self-RAG 与回答校验，降低错误自信回答。
4. 再推进 Tool Use、SSE、引用高亮和自动评测。

## 8. 非目标

当前阶段不建议优先投入以下方向：

- 面向所有格式一次性做完整 parser 覆盖。
- 为 Agent 开放无边界通用工具集。
- 过早追求多智能体复杂编排，而忽略单链路质量。
- 在没有标准评测集前频繁切换模型与策略。

## 9. 交付物清单

每个阶段至少应交付以下内容：

- 设计文档
- 代码实现
- 配置项说明
- 调试接口或可视化面板
- 基准测试或评测报告
- 回滚方案与降级开关

## 10. 结论

下一阶段的核心，不是继续堆叠更多“点状功能”，而是围绕“结构化输入 -> 高质量检索 -> 可验证生成 -> 闭环评测”建立持续演进的系统能力。完成这条路线后，项目才真正具备向工业级 Agentic RAG 平台扩展的基础。
